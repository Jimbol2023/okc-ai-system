# Anti-Runtime Drift Safeguards

## Purpose

Runtime drift occurs when planning, readiness, certification, authorization, or stabilization language begins functioning as operational permission.

## Safeguards

The system must guard against:

- readiness becoming permission;
- certification becoming authorization;
- authorization becoming activation;
- activation becoming autonomous execution;
- provider readiness becoming onboarding;
- adapter design becoming implementation;
- module request becoming approval;
- treasury review becoming spend authority;
- governance containment becoming runtime control.

## Detection Questions

Reviewers must ask:

- Did any doc create execution authority?
- Did any boundary become ambiguous?
- Did any runtime path become implied?
- Did any provider or adapter gain authority?
- Did any module bypass the shared core?
- Did any treasury boundary weaken?

## Default Response

Runtime drift is resolved by denial, not interpretation.
